# Admin Shell Curation Design

**Context**

当前管理端已经具备完整的后台能力：

- `Dashboard` 展示索引状态与任务队列。
- `Search` 支持文本检索与相似图检索。
- `Images` 提供图片浏览、筛选、批量操作与详情查看。
- `Tags` / `Categories` 支持标签与分类维护，并已接入图片明细页。
- `Login` 提供最小可用的鉴权入口。

现状问题不在功能缺失，而在界面语言仍然接近通用后台模板：

- 全局壳层偏浅色、卡片化，图片没有成为主角。
- 页面层级主要依赖边框和 `Card`，缺少策展式浏览节奏。
- 搜索、图库、标签、分类虽然服务于“看图与组织图”，视觉体验却仍是表单后台。
- 登录页缺少产品气质，无法建立明确的工具感。

用户已确认本次改造方向为：

- 整个后台统一重设计，而不是单页改皮肤。
- 风格采用“偏图片策展，但保持后台可操作性”的中间路线。
- 默认主题可以直接切换到深色策展风格。

**Goals**

- 将整个管理端重塑为深色策展工作台，让图片检索与浏览成为主舞台。
- 保留现有路由、业务能力、接口契约与操作密度，不做后端重构。
- 以共享壳层、主题变量、页面结构重排为主，减少无必要的组件推倒重写。
- 让 `Dashboard`、`Search`、`Images`、`Tags`、`Categories`、`Login` 在视觉上形成同一体系。
- 通过克制而明确的动效增强层级和空间感，而不是用装饰性动画堆砌。

**Non-Goals**

- 不修改 HTTP API、MCP 工具或运行时服务。
- 不改变标签、分类、图片、检索的业务语义。
- 不新增主题切换、用户偏好、个性化布局等扩展功能。
- 不引入重量级新 UI 框架或重写所有基础 shadcn 组件。

**Visual Thesis**

把管理端做成“暗房中的策展工作台”：整体空间深、静、低反差，图片和结果区域被照亮，铜色/琥珀色作为唯一主要强调色，用来提示当前路径、主要动作和活动状态。

设计原则：

- 品牌和当前工作区必须在首屏立即可识别。
- 图片、结果、浏览路径优先于卡片和数据板块。
- 通过版式、密度、留白和层次组织内容，而不是一块块白卡片。
- 默认只保留一个强调色，避免后台工具感被多色状态污染。
- 所有页面都应当像同一套策展系统中的不同工作模式，而不是若干独立页面。

**Content Plan**

整套后台采用统一的四段式节奏，但映射到运营工具语言：

1. Hero / Shell
   - 左侧是策展式目录导航。
   - 顶部是当前页面上下文、关键状态、主要动作。
   - 第一屏先建立“我现在在整理图库”这一感受。
2. Support
   - `Dashboard` 显示索引覆盖、任务活动、模型配置、最近系统变化。
3. Detail
   - `Search` 与 `Images` 成为主要视觉舞台。
   - `Tags` 与 `Categories` 提供组织图片所需的轻量编辑界面。
4. Final CTA
   - 每个页面保留一个清晰主动作，例如开始检索、发起更新、创建标签、创建分类。

**Interaction Thesis**

本次交互只做三类有意义的动效：

- 页面进入：外层壳层先稳定出现，内容区做轻微上移和淡入。
- 空间切换：导航高亮、面板展开、对话框出现时体现方向感与层次感。
- 图片反馈：结果项、图库项、详情查看使用轻微缩放、亮度与阴影变化，强化“选片”感。

约束：

- 动效时长短、节制、全站一致。
- 在移动端依然要稳定，不依赖复杂滚动编排。
- 若某个动效不能改善理解或操作，直接删除。

**Recommended Approach**

采用“策展工作台型”方案：

1. 保留现有 React 路由、数据流和页面职责。
2. 重做共享壳层与全局设计令牌，建立深色策展视觉体系。
3. 优先改造 `Dashboard`、`Search`、`Login` 的结构与节奏。
4. 对 `Images`、`Tags`、`Categories` 做布局和信息层级重排，让它们融入同一系统。
5. 尽量在页面布局层消解卡片感，而不是全面重写基础 UI 组件。

为什么不选另外两条路：

- “保守升级型”无法让图片真正成为中心，成品会像普通后台换肤。
- “激进画廊型”会损害标签维护、分类树、批量操作等运营效率。

**Information Architecture**

全局结构调整如下：

- 左侧导航：
  - 品牌名 `Image Search` 放大并成为主识别点。
  - 导航项减少按钮感，更像目录索引。
  - 当前路由用高亮轨道、微弱背景与强调色文字共同指示。
- 顶部上下文栏：
  - 显示页面标题、简短说明、关键元信息、主要操作。
  - 不再让每个页面自己单独从零组织标题和按钮。
- 主内容区：
  - 统一最大宽度与内部节奏，但对图片页保持更开放的画布。
  - 常规管理页使用分区布局，不默认包一层大卡片。

**Page-Level Design**

`Dashboard`

- 从 KPI 卡片页改成“图库总览台”。
- 索引覆盖作为主视觉信息块，使用更强的比例图形与大号数字。
- 任务列表改为时间流或状态带，强调当前系统正在发生什么。
- 嵌入模型、版本、图片总量等信息压缩为辅助元数据。
- 主要动作仅保留 `Incremental Update` 和 `Full Rebuild`，且视觉权重清晰区分。

`Search`

- 作为最重要页面处理。
- 搜索输入区与检索模式选择整合进顶部上下文区域，不再分裂成两张卡。
- 结果区放大，优先展示缩略图、相似度、标签与尺寸。
- 查询条件、结果计数、检索类型保持可扫读，但不喧宾夺主。
- 打开图片详情时，体验更接近看片灯箱，而不是传统后台弹窗。

`Images`

- 保持现有浏览与批量能力，但视觉上更接近图库浏览器。
- 工具栏、过滤器、视图切换、批量动作整合成统一的操作条。
- 列表与网格视图使用更克制的边框与背景，让图片内容成为主信息。

`Tags`

- 将“创建标签”和“标签列表”整合为轻量编排界面。
- 标签项像索引词，不像被装进卡片的小对象。
- 批量模式与删除流程保留，但视觉上从“按钮工具条”改为更安静的编辑状态。

`Categories`

- 保留分类树与批量能力。
- 弱化树组件的后台感，让层级、选中、展开状态更容易扫读。
- 创建、编辑、移动、删除等动作仍是核心，但页面应更像“结构编排”而非表单集。

`Login`

- 从居中小卡片改为完整首屏。
- 左侧或背景建立产品氛围，右侧保留简洁登录表单。
- 文案只需说明它是图像检索后台，不引入营销式口号堆砌。

- `src/image_search_mcp/web/src/pages/SearchPage.tsx`
- `src/image_search_mcp/web/src/pages/LoginPage.tsx`

次级修改文件：

- `src/image_search_mcp/web/src/pages/ImagesPage.tsx`
- `src/image_search_mcp/web/src/pages/TagsPage.tsx`
- `src/image_search_mcp/web/src/pages/CategoriesPage.tsx`
- `src/image_search_mcp/web/src/components/ImageBrowser.tsx`
- `src/image_search_mcp/web/src/components/SearchResultCard.tsx`
- `src/image_search_mcp/web/src/components/ImageModal.tsx`
- `src/image_search_mcp/web/src/components/CategoryTree.tsx`
- `src/image_search_mcp/web/src/components/FilterBar.tsx`

实现原则：

- 用全局令牌先统一颜色、边界、间距、阴影、字体。
- 通过页面布局与共享展示组件降低卡片感。
- 只在必要时增加少量展示型封装组件，如页头、状态条、策展式统计块。

**Design Tokens**

建议引入或重定义以下视觉令牌：

- 背景分层：
  - 主背景：深灰黑
  - 次级面：略亮的炭灰
  - 浮层：更高亮度但低饱和的面
- 强调色：
  - 铜色 / 琥珀色，用于主动作、激活状态、关键数值
- 文本层级：
  - 主文本高对比
  - 次级说明中等对比
  - 元信息低对比但仍满足可读性
- 边界：
  - 大量使用低对比描边或内发光替代厚重边框
- 圆角：
  - 保留中等圆角，避免 SaaS 卡片感过强

**Responsive Behavior**

- 桌面端采用固定侧边导航与宽画布内容区。
- 平板与小桌面端保持导航可读，内容区优先保证图片宽度。
- 移动端允许顶部导航或抽屉式侧栏，但不为此单独设计第二套视觉语言。
- 搜索和图库页在小屏下仍需保证首屏可直接看到输入或图片结果，而不是先滚过大段说明区。

**Error Handling and Empty States**

- 空状态依然保持克制，不做插画式占位。
- 搜索无结果时强调查询上下文和下一步动作，而不是只显示“0 results”。
- 加载状态采用骨架或低频闪烁，避免高噪声动画。
- 登录错误、任务失败、检索失败等反馈沿用现有 toast 机制，但视觉主题需统一到新系统。

**Testing Strategy**

视觉重构后的验证重点：

- 路由导航测试仍通过，确保 `Layout` 结构变化不破坏现有访问路径。
- 登录流程测试仍通过，确保登录页改造不影响认证提交。
- 图片、标签、分类相关测试重点验证现有交互和文本未因结构改造失效。
- 新增或更新前端测试以覆盖：
  - 新导航标题与主区域语义结构
  - `Dashboard` 的主统计与主要操作
  - `Search` 的统一检索入口与结果区域

**Risks**

- 当前页面大量依赖 `Card` 组件，若一次性重写过多基础组件，回归风险会显著上升。
- 深色主题下如果对比控制不佳，标签、元信息、表单状态会变得难扫读。
- 搜索和图库页若过度追求画廊感，可能牺牲批量操作效率。
- 登录页如果视觉表现过强，可能破坏后台工具的一致气质。

**Mitigations**

- 先改共享壳层和主题变量，再逐页推进。
- 优先在布局层降卡片感，必要时只对少数组件做局部样式扩展。
- 每改完一个关键页面就运行针对性前端测试，避免集中回归。
- 对比度、焦点态、可点击区域保持运营工具标准，不为审美牺牲可操作性。

**Acceptance Criteria**

- 整个后台默认进入深色策展风格，而不是浅色通用后台。
- 首屏能够立即识别 `Image Search` 品牌和当前工作区域。
- `Search` 和 `Images` 的图片主导感显著提升。
- `Dashboard` 从传统卡片页转为更有层级的图库总览台。
- `Tags` / `Categories` 保留管理效率，同时融入统一视觉语言。
- `Login` 具备完整首屏气质，与后台主体风格一致。
- 现有路由、鉴权、搜索、标签、分类、图片操作流程不回退。
