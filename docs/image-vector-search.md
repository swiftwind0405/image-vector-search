# QNAP 上供 Agent 调用的本地图片语义检索 MCP 服务方案

## 1. 文档目标

本文档定义一个运行在 QNAP 上、供局域网内单个 agent 通过 MCP 调用的本地图片语义检索服务。

目标是让 agent 能根据自然语言语义，从本地图片目录中检索出最匹配的图片，并返回当前有效的文件路径。

本方案强调以下约束：

- 图片规模为几万张，当前不追求高并发和分布式能力
- 向量化通过付费 API 完成，当前使用 Jina，不在本地运行大模型
- 需要尽量复用已有 embedding，避免因为路径变化或误操作导致重复付费
- 日常以手动触发增量更新为主，特殊情况下才执行手动全量重建
- 对 agent 暴露的唯一调用入口是 MCP，不直接暴露向量数据库
- 服务部署在 QNAP 上，代码托管在 GitHub，通过 GitHub Actions 构建镜像并发布到 GHCR

---

## 2. 范围与非目标

### 2.1 范围

本项目覆盖以下能力：

- 通过文本语义检索本地图片
- 通过示例图片检索相似图片
- 手动触发图片索引的增量更新
- 手动触发图片索引的全量重建
- 在文件移动、重命名、目录调整后尽量复用已有 embedding
- 只返回每张图片的一个当前有效路径

### 2.2 非目标

当前版本不覆盖以下内容：

- 面向公网或多租户的访问能力
- 自动定时扫描、自动同步、事件驱动监听
- 多 agent 并发共享检索服务的复杂调度
- 完整的标签管理、人工标注、OCR、EXIF 深度解析
- 面向 UI 的图片预览系统

---

## 3. 关键设计决策

### 3.1 采用 MCP 作为 agent 接入层

agent 最终通过 MCP tools 调用本服务。对 agent 而言，服务暴露的是一组语义清晰的工具，而不是脚本命令、数据库连接或底层存储细节。

### 3.2 采用 Milvus Lite，而不是独立部署 Milvus Standalone

当前场景只有局域网内单个 agent 使用，规模也不重，因此不采用独立的 Milvus 服务进程，而采用嵌入式的 Milvus Lite。

部署目标为 x86 架构的 QNAP NAS，Milvus Lite 对此架构有完整支持。

这样做的原因：

- 部署更简单，QNAP 上只需要维护一个主服务容器
- 数据以本地文件形式持久化，符合当前轻量场景
- 不需要额外暴露向量数据库端口
- 后续如果规模增加，再演进到独立 Milvus 也有清晰迁移路径

### 3.3 引入 SQLite 作为图片元数据索引

Milvus Lite 负责向量检索，但不负责承担路径管理、文件状态追踪和增量同步逻辑。

因此需要额外引入 SQLite 保存图片的元数据，用于：

- 记录图片内容哈希
- 维护当前有效路径
- 跟踪文件大小、修改时间、最后扫描时间
- 支撑路径变更、重复文件、失效文件的管理

### 3.4 图片身份以内容哈希为准，而不是文件路径

文件路径不是稳定标识。图片移动、重命名、整理目录时，路径会变化，但图片内容可能完全相同。

因此系统以 `content_hash`（SHA-256，对文件完整内容计算）作为图片的稳定身份标识，路径只作为可变元数据保存。

这项设计直接决定了以下行为：

- 路径变化时，优先更新元数据，不重算 embedding
- 同一张图出现在多个路径时，只保留一个主路径用于返回
- 只有真正的新图片内容才会触发新的向量化调用

## 4. 总体架构

```text
┌────────────────────────────┐
│ 局域网内 Agent             │
│ 通过 MCP 调用工具          │
└──────────────┬─────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│ QNAP 容器: image-search-mcp         │
│                                     │
│ 1. MCP Server                       │
│ 2. 检索与索引服务逻辑               │
│ 3. Milvus Lite                      │
│ 4. SQLite 元数据数据库              │
└──────────────┬──────────────┬───────┘
               │              │
               │              └──────────────┐
               │                             │
               ▼                             ▼
      本地图片目录                    Jina Embedding API
      /data/images                    文本/图片向量化
```

核心原则：

- agent 只调用 MCP tools
- MCP 服务是唯一业务入口
- Milvus Lite 和 SQLite 都是 MCP 服务的内部依赖
- 本地图片目录只读挂载
- 所有敏感参数通过环境变量注入

## 5. 部署拓扑

### 5.1 部署方式

通过 docker-compose 部署为一个主应用容器：

- 容器名建议：`image-search-mcp`
- 所有配置参数通过环境变量注入

该容器内部包含：

- MCP 服务进程（使用 Streamable HTTP 传输协议，供局域网内 agent 远程调用）
- Milvus Lite 本地数据库文件
- SQLite 元数据数据库文件

### 5.2 目录挂载建议

QNAP 上建议挂载两类目录：

- 图片目录：例如 `/share/Photos`，挂载到容器内 `/data/images`
- 索引目录：例如 `/share/Container/image-search-mcp/index`，挂载到容器内 `/data/index`

其中：

- `/data/images` 只读挂载
- `/data/index` 用于持久化 Milvus Lite 和 SQLite 数据文件

### 5.3 镜像发布

代码存放于 GitHub 仓库，通过 GitHub Actions 构建 Docker 镜像并发布到 GHCR。

由于部署目标是 QNAP，镜像应保持公开可拉取，敏感参数不进入镜像，而是通过环境变量注入。

## 6. 存储设计

### 6.1 Milvus Lite

Milvus Lite 用于存储图片向量与向量检索所需的最小字段。

建议 collection 逻辑上至少包含：

- `content_hash`：图片内容哈希，作为稳定身份
- `embedding`：图片向量

Milvus Lite 的职责只有两个：

- 根据文本或图片向量执行相似度检索
- 返回命中的 `content_hash`

Milvus Lite 不承担以下职责：

- 当前路径是否有效
- 文件是否被删除
- 文件路径是否发生变化
- 重复图片如何选择主路径

这些都由 SQLite 负责。

### 6.2 SQLite 元数据数据库

SQLite 用于管理图片元数据与文件状态。

需要维护两张表：`images` 表和 `image_paths` 表。

#### `images` 表

记录图片的内容身份与元数据，字段包括但不限于：

- `content_hash`：图片稳定身份（SHA-256），主键
- `canonical_path`：当前返回给 agent 的主路径
- `file_size`：文件大小
- `mtime`：文件修改时间
- `mime_type`：MIME 类型
- `width`：图片宽度
- `height`：图片高度
- `last_seen_at`：最近一次扫描到该图片的时间
- `is_active`：该图片当前是否仍存在于可检索范围内

#### `image_paths` 表

记录同一张图片的所有已知路径，用于主路径失效时自动切换。字段包括但不限于：

- `id`：自增主键
- `content_hash`：关联 `images` 表
- `path`：文件路径（容器内路径）
- `last_seen_at`：该路径最近一次被扫描到的时间
- `is_active`：该路径当前是否仍有效

### 6.3 路径管理策略

系统不以”路径”为结果单位，而以”图片内容”作为结果单位。对同一个 `content_hash`，检索只返回一个 `canonical_path`。

但通过 `image_paths` 表记录所有已知路径，系统可以实现以下能力：

- 主路径失效时，自动从 `image_paths` 中选择其他仍然有效的路径作为新的 `canonical_path`
- 在增量更新报告中可统计重复数量，便于后续清理

主路径选择规则：

- 优先保留已存在的主路径
- 若主路径失效，从 `image_paths` 中选择一个仍然活跃的路径
- 若无活跃路径，标记该图片为非活跃

## 7. MCP 工具设计

MCP 是本系统对 agent 暴露的唯一接口。建议至少提供以下工具。

### 7.1 `search_images`

用途：

- 根据文本语义检索本地图片

输入参数建议：

- `query`：查询文本
- `top_k`：返回数量
- `min_score`：最小相似度阈值
- `folder`：可选的目录范围过滤（容器内路径）

输出结果建议：

- `content_hash`
- `path`（容器内路径）
- `score`
- `width`
- `height`
- `mime_type`

### 7.2 `search_similar`

用途：

- 根据一张已有图片检索相似图片

输入参数建议：

- `image_path`：查询图片路径（容器内路径）
- `top_k`：返回数量
- `min_score`：最小相似度阈值
- `folder`：可选的目录范围过滤（容器内路径）

输出结构与 `search_images` 一致。结果中自动排除查询图片本身。

### 7.3 `update_index_incremental`

用途：

- 手动触发增量更新（同步阻塞，调用后等待完成再返回结果）

适用场景：

- 新增了部分图片
- 调整了部分目录
- 重命名或移动了图片文件
- 删除了部分图片

返回结果建议包含：

- 扫描文件数
- 新增图片数
- 复用已有 embedding 数
- 路径更新数
- 失效图片数
- 跳过图片数
- 错误数

### 7.4 `rebuild_index_full`

用途：

- 手动触发全量重建（同步阻塞，调用后等待完成再返回结果）

适用场景：

- 索引状态严重不一致
- 目录结构发生大范围调整
- 需要执行大版本迁移
- 需要重新生成元数据结构

全量重建不等于“无脑清空后重新向量化所有图片”。

建议默认行为为：

- 全量重扫文件系统
- 重建 SQLite 中的文件元数据视图
- 能复用已有 `content_hash` 和 embedding 的尽量复用
- 只对真正缺失 embedding 的图片调用向量化 API

只有在明确要求彻底重置时，才执行“删除全部向量并重新生成”的破坏性操作。

### 7.5 `get_index_status`

用途：

- 查看当前索引状态，帮助 agent 判断是否需要触发更新

返回结果建议包括：

- 当前图片总数
- 活跃图片数
- 非活跃图片数
- 向量条目数
- 最近一次增量更新时间
- 最近一次全量重建时间
- 最近一次错误摘要

### 7.6 调用示例

以下为 agent 调用 MCP 工具的典型交互示例。

#### 文本搜图

请求：

```json
{
  "tool": "search_images",
  "arguments": {
    "query": "海边的日落",
    "top_k": 3
  }
}
```

响应：

```json
{
  "results": [
    {
      "content_hash": "a1b2c3d4...",
      "path": "/data/images/2024/vacation/sunset_beach.jpg",
      "score": 0.92,
      "width": 4032,
      "height": 3024,
      "mime_type": "image/jpeg"
    },
    {
      "content_hash": "e5f6g7h8...",
      "path": "/data/images/2023/summer/ocean_dusk.jpg",
      "score": 0.87,
      "width": 3840,
      "height": 2160,
      "mime_type": "image/jpeg"
    }
  ]
}
```

#### 以图搜图

请求：

```json
{
  "tool": "search_similar",
  "arguments": {
    "image_path": "/data/images/2024/vacation/sunset_beach.jpg",
    "top_k": 5,
    "folder": "/data/images/2023"
  }
}
```

响应结构与 `search_images` 一致，结果中不包含查询图片本身。

#### 增量更新

请求：

```json
{
  "tool": "update_index_incremental",
  "arguments": {}
}
```

响应：

```json
{
  "scanned": 12500,
  "added": 42,
  "reused": 12400,
  "path_updated": 15,
  "deactivated": 3,
  "skipped": 40,
  "errors": 0
}
```

## 8. 核心流程设计

### 8.1 文本搜图流程

1. agent 调用 `search_images`
2. MCP 服务将文本发送给 Jina 生成查询向量
3. 使用查询向量在 Milvus Lite 中检索最相近的 `content_hash`
4. 使用 `content_hash` 到 SQLite 查询当前有效的 `canonical_path`
5. 过滤掉无效、已删除或不在目标目录范围内的记录
6. 将结果返回给 agent

### 8.2 以图搜图流程

1. agent 调用 `search_similar`
2. MCP 服务读取指定图片文件，计算其 `content_hash`
3. 调用 Jina 生成图片向量
4. 在 Milvus Lite 中检索最相近的 `content_hash`
5. 从结果中排除查询图片自身的 `content_hash`
6. 从 SQLite 解析出当前有效路径
7. 返回结果给 agent

### 8.3 增量更新流程

默认更新方式是手动触发增量更新。

建议的大致逻辑如下：

1. 扫描目标目录中的文件清单
2. 对每个文件，根据 `mtime`（修改时间）和 `file_size`（文件大小）判断是否可能发生变化；对新增文件或疑似变化文件计算 `content_hash`
3. 使用 `content_hash` 在 SQLite 中判断该图片是否已存在
4. 若 `content_hash` 已存在：
   - 在 `image_paths` 表中记录当前路径
   - 按需更新 `canonical_path`、`mtime`、`last_seen_at` 等元数据
   - 不重新生成 embedding
5. 若 `content_hash` 不存在：
   - 调用 Jina 生成 embedding
   - 写入 Milvus Lite
   - 写入 SQLite `images` 表和 `image_paths` 表
6. 对扫描中未再次出现的旧路径：
   - 在 `image_paths` 中标记为非活跃
   - 若某 `content_hash` 的所有路径均失效，将 `images` 中对应记录标记为非活跃
7. 输出更新摘要

该流程的核心目标是：优先复用 embedding，最小化付费 API 调用。

### 8.4 全量重建流程

全量重建只在特殊需求下手动触发。

建议的大致逻辑如下：

1. 对目标目录重新执行完整扫描
2. 重建 SQLite 的文件视图与活动状态
3. 尽可能复用 Milvus Lite 中已存在的 `content_hash`
4. 仅对缺失向量的图片重新调用 Jina
5. 对过期或无效的元数据进行统一清理
6. 输出完整重建报告

该流程的目标是：重建索引一致性，而不是制造不必要的重算成本。

## 9. 路径变化与重复文件处理

### 9.1 路径变化

路径变化不会改变图片语义，但会影响结果可用性。

如果系统只按路径存储，那么文件移动或重命名后，检索结果可能返回失效路径。

本方案通过 `content_hash` 解决这一问题：

- 同一图片只要内容不变，就视为同一个对象
- 文件路径变化时，优先更新 SQLite 中的 `canonical_path`
- 不重新生成 embedding

### 9.2 文件删除

若某图片在最近一次扫描中未再次出现，应按策略处理：

- 默认标记为 `is_active = false`
- 查询时不返回非活跃图片
- 在后续全量重建或清理操作中可进一步移除无效元数据

### 9.3 重复文件

如果同一张图片以多个路径存在：

- 所有路径均记录在 `image_paths` 表中
- `images` 表只维护一个 `canonical_path`，检索结果只返回这一条主路径
- 主路径失效时，自动从 `image_paths` 中选择仍然活跃的路径替补

主路径选择规则：

- 优先保留已存在的主路径
- 若主路径失效，从 `image_paths` 中按路径字典序选择第一个活跃路径
- 若无活跃路径，标记该图片为非活跃

## 10. 配置与安全要求

### 10.1 环境变量

敏感参数必须通过环境变量注入，例如：

- `JINA_API_KEY`

普通配置也建议通过环境变量控制，例如：

- 图片根目录
- 索引目录
- 默认返回数量
- 最大返回数量
- 最低分数阈值

### 10.2 路径约定

所有 MCP 工具中涉及的路径参数（`folder`、`image_path`）以及返回结果中的 `path`，均为容器内路径（例如 `/data/images/2024/photo.jpg`）。

宿主机路径到容器内路径的映射通过 docker-compose 的 volume 挂载配置完成，MCP 服务本身不感知宿主机路径。

### 10.3 错误处理

当 Jina API 调用失败时，默认采用以下策略：

- 自动重试最多 3 次，每次间隔指数退避
- 重试仍失败的图片跳过处理，记录到错误日志
- 在增量更新或全量重建的返回结果中，通过 `错误数` 字段体现失败情况
- 不因单张图片的向量化失败而中断整个更新流程

### 10.4 访问边界

当前只计划服务于局域网内单个 agent，因此应遵循以下原则：

- 不对外暴露 Milvus Lite 或 SQLite
- 对 agent 暴露的仅是 MCP 服务
- 图片目录使用只读挂载
- 索引目录单独持久化，便于备份与迁移

### 10.5 成本控制

由于 embedding 依赖付费 API，必须明确以下控制原则：

- 增量更新优先
- 路径变化不触发重新向量化
- 全量重建默认复用已有 embedding
- 只有新图片内容或明确强制重算时，才调用向量化 API

## 11. 管理操作建议

虽然对 agent 暴露的是 MCP tools，但从运维角度，系统仍应具备清晰的管理动作。

建议至少支持以下四类操作：

- 语义搜图
- 以图搜图
- 手动增量更新
- 手动全量重建

从产品角度看，推荐将“手动增量更新”作为日常标准操作，将“手动全量重建”视为维护级操作，而不是常规路径。

---

## 12. 推荐方案总结

基于当前约束，推荐的最终方案为：

- 对 agent 暴露 MCP 服务
- 服务运行在 QNAP 的单个主容器中
- 使用 Jina API 进行文本和图片向量化
- 使用 Milvus Lite 保存向量索引
- 使用 SQLite 保存图片元数据和当前有效路径
- 使用 `content_hash` 作为图片稳定身份
- 默认采用手动增量更新
- 特殊情况下再执行手动全量重建

这套方案的核心收益是：

- 部署轻量
- 运行简单
- 成本可控
- 路径变化可处理
- 能为后续演进到更重的独立向量数据库方案保留空间

---

## 13. 后续实现建议

进入实现阶段时，建议优先按以下顺序推进：

1. 先实现最小可用的 MCP 搜图能力
2. 再补增量更新能力
3. 再补全量重建能力
4. 最后补状态查询、错误报告和运维细节

当前文档仅定义需求、架构和大致实现逻辑，不展开代码实现细节。
